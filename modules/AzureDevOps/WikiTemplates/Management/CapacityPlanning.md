## Capacity Planning Factors

| Factor | % of Time | Rationale |

| **Productive Development** | 80% | Industry standard for sustainable pace |

- **Junior Developers**: 20-30% in learning/ramp-up
- **On-call Engineers**: 10-20% reserved for incidents

## Capacity Planning Best Practices

### ‚úÖ DO
1. **Plan conservatively**: Use 70-80% of theoretical capacity
2. **Track actuals**: Compare planned vs. actual to improve forecasting
3. **Account for interruptions**: Reserve 10-15% for unplanned work
4. **Update regularly**: Refresh capacity data each sprint
5. **Include non-dev time**: Meetings, code reviews, learning
6. **Communicate early**: Flag capacity issues before sprint starts
7. **Balance workloads**: Distribute work according to capacity
8. **Factor in complexity**: Some work items need more focused time

### ‚ùå DON'T
1. **Don't assume 100% productivity**: Unrealistic and leads to burnout
2. **Don't ignore patterns**: Historical data guides future planning
3. **Don't overcommit**: Better to under-promise and over-deliver
4. **Don't forget holidays**: Check calendar for upcoming time off
5. **Don't skip buffer time**: Always reserve capacity for unknowns
6. **Don't ignore team health**: Sustainable pace > maximum velocity

## Capacity Adjustment Scenarios

### Scenario 1: New Team Member Joins
- **Week 1-2**: 25% productive capacity (onboarding, learning)
- **Week 3-4**: 50% productive capacity (supervised work)
- **Week 5-8**: 75% productive capacity (gaining independence)
- **Week 9+**: 100% productive capacity (full contributor)

### Scenario 2: Team Member Departure
- **2 weeks before**: 50% capacity (documentation, knowledge transfer)
- **Last week**: 25% capacity (handoff, transition)
- **After departure**: Redistribute work or hire

### Scenario 3: Major Production Incident
- **During incident**: 50% capacity reduction for on-call team
- **Post-incident (1-2 days)**: 25% reduction (RCA, fixes)
- **Recovery (week)**: Return to normal capacity

### Scenario 4: Holiday/Vacation Period
- **Week before holiday**: 90% capacity (people wrapping up)
- **Holiday week**: 60-70% capacity (reduced staffing)
- **Week after holiday**: 85% capacity (people catching up)

## Capacity Planning Tools

### Azure DevOps Capacity Feature
Navigate to: **Boards ‚Üí Sprints ‚Üí Capacity**

Set capacity for each team member:
1. Enter days off
2. Set activity allocation (Development, Testing, Design, etc.)
3. Specify hours per day
4. Azure DevOps calculates remaining capacity automatically

### Capacity Dashboard Widgets
Add these widgets to your team dashboard:
- **Sprint Capacity**: Shows planned vs. remaining capacity
- **Sprint Burndown**: Tracks work remaining over time
- **Sprint Velocity**: Historical velocity trends
- **Team Members**: Quick view of who's on the team

### Formulas & Calculations
```powershell
# Total Capacity
$totalCapacity = $teamSize √ó $sprintDays √ó $hoursPerDay √ó $productivityFactor

# Net Capacity
$netCapacity = $totalCapacity - $plannedAbsences - $meetingTime

# Expected Story Points
$expectedSP = $netCapacity √ó $spPerHourRatio

# Example for 8-person team, 10-day sprint
$totalCapacity = 8 √ó 10 √ó 8 √ó 0.80 = 512 hours
$netCapacity = 512 - 32 (absences) - 64 (meetings) = 416 hours
$expectedSP = 416 √ó 0.06 = ~25 story points
```
